what are ansible adhoc commands?
it is ansible command line to perform one time tasks or
quick tasks when we don't have to write a playbook.

why use adhoc commands?

it useful for 
* checking connectivity, 
* running one-time tasks,
* quick troubleshooting
* gathering information.

basic sysntax: ansible <host-pattern> -m <module> -a "<arguments>"

examples: 

ping all servers

 ansible all -i inventory.ini -m ping   --> Checks if all nodes are reachable

check uptime of servers:

ansible all -i inventory.ini -m command -a "uptime" 

Install a Package on RHEL:

ansible all -i inventory.ini -m yum -a "name=httpd state=present" --become

start a service:

ansible all -i inventory.ini -m service -a "name=httpd state=started" --become

Remove the package:

Install a Package on RHEL:

ansible all -i inventory.ini -m yum -a "name=httpd state=absent" --become

create a user:
ansible all -i inventory.ini -m user -a "name=testuser state=present" --become

check user is created or not:
ansible all -i inventory.ini -m command -a "grep testuser /etc/passwd"

Run shell command:
ansible all -i inventory.ini -m shell -a "df -h"

gather system information:
ansible all -i inventory.ini -m setup

Restart a service:
ansible webservers -m service -a "name=httpd state=restarted" --become

how to copy files:

echo "Hello from Ansible" > file.txt

ansible all -i inventory.ini -m copy -a "src=file.txt dest=/tmp/file.txt"

Verify on remote nodes:

ansible all -i inventory.ini  -m shell -a "cat /tmp/file.txt"




03.nginx.yaml:  install nginx and start the nginx
=== ===========

- name: nginx install and run
  hosts: frontend
  become: yes # sudo access
  tasks:
  - name: install nginx server
    ansible.builtin.package:
      name: nginx
      state: present
  
  - name: start nginx
    ansible.builtin.service:
      name: nginx
      state: started
      enabled: yes
	  
ansible-playbook -i inventory.ini -e ansible_user=ec2-user -e ansible_password=DevOps321 03.nginx.yaml

12.vars-preference.yaml:
============ ==== ====

course.yml -->

inventory.ini
GREETING="Hello from Inventory"

---
- name: understand vars preference
  hosts: local
  connection: local
  vars:
     GREETING: "Hello from PLAY"
  vars_files:
   - course.yaml
  vars_prompt:
   - name: GREETING
     prompt: Please enter the greeting
     private: false
  tasks:
  - name: print the greeting
     vars:
       GREETING: "Hello from TASK"
    ansible.builtin.debug:
      msg: "{{ GREETING }}"
	  

first we can run through args:

ansible-playbook -i inventory.ini 12.vars-preference.yaml -e "GREETING=Hello-from-args"


first preperence is command line arguments.

next we run the playbook: ansible-playbook -i inventory.ini 12.vars-preference.yaml

please enter the greeting: hello from prompt --> its asking prompt but the next preference is task.

so second preference is task level.

comment the task play lines:
 #vars:
       #GREETING: "Hello from TASK"
	   
after run the playbook: ansible-playbook -i inventory.ini 12.vars-preference.yaml

Enter the greeting: hello from prompt
but it take file.

third preference is filelevel variables.

 #  - course.yaml

enter the prompt: hello from prompt

fourth preference is prompt.
so we can comment the prompt lines:

  #vars_prompt:

14-conditions.yaml:
===== ==============

- name: conditions demo
  hosts: local
  connection: local
  vars:
    NUMBER: 78
  tasks:
  - name: print if the number is less than or equal to 100
    ansible.builtin.debug:
      msg: "Given number {{ NUMBER }} is less than or equal to 100"
    when: NUMBER <= 100

  - name: print if the number is greater than 100
    ansible.builtin.debug:
      msg: "Given number {{ NUMBER }} is greater than 100"
    when: NUMBER > 100
